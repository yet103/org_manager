## 1. 最初に伝えるべき5項目

AIに開発を依頼する際、**冒頭で以下を明確にする**と手戻りが激減する：

```
① ゴール    — 何を作るか（1行で）
② 技術制約  — 使う/使わない技術
③ 配布方法  — どう動かすか
④ データ構造 — 何を管理するか
⑤ UI構成    — 画面のレイアウト
```

### 理想的な初期指示の例

```
組織図を作成・編集するWebアプリを作ってください。

技術: HTML/CSS/JSのみ。フレームワーク不使用。サーバー不要。
動作: index.htmlをダブルクリックで起動。データはlocalStorageに自動保存。
データ: 人物（名前,所属,色,座標）、領域（部署の矩形）、コネクタ（関係線）
UI: 左=人物一覧、中央=Canvas描画、右=プロパティパネル、上=ツールバー
操作: ドラッグ&ドロップ、ズーム、Undo/Redo、範囲選択
```

**これだけで骨格の80%が決まる。** 今回はこれを数回に分けて伝えたため、都度修正が発生した。

## 2. 機能は「層」で指示する

### ❌ やりがちな失敗：全機能を一度に列挙

```
検索、エクスポート、CSV取込、ダークモード、注釈、ショートカット、
印刷、共有URL、色分け、人数バッジ...全部やって
```
→ AIが混乱、中途半端な実装になりやすい

### ✅ ベストプラクティス：レイヤーごとに依頼

```
Phase 1: コア機能（描画、配置、保存）
Phase 2: 編集機能（Undo、範囲選択、コネクタ）
Phase 3: データ管理（JSON保存/読込、CSV取込）
Phase 4: UX改善（検索、ショートカット、ダークモード）
Phase 5: 出力・共有（PNG出力、印刷、共有URL）
```

各Phaseを**完了→動作確認→次のPhase**で進めると品質が安定する。

## 3. 指示のフォーマット

### 効果が高かった指示パターン

| パターン | 例 | 効果 |
|----------|-----|------|
| **具体的なUI指示** | 「領域名の横に所属人数をバッジ表示」 | 一発で正確に実装される |
| **既存UIとの対比** | 「ExcelのCtrl+Clickと同じ複数選択」 | 認識のずれがない |
| **否定形の制約** | 「サーバー不要にできませんか？」 | 設計方針が明確になる |
| **一覧表での依頼** | ショートカットキーの表を渡す | 漏れなく実装される |

### 効果が低かった指示パターン

| パターン | 例 | 問題点 |
|----------|-----|--------|
| **曖昧な依頼** | 「もっと使いやすく」 | AIが何をすべきか判断できない |
| **巨大な一括依頼** | 「#9〜#11を全部実装して」 | 品質にムラが出る |
| **後出しの制約** | 機能実装後に「サーバーなしで」 | 大幅な作り直しが発生 |

## 4. 今回の開発で発生した手戻りと対策

| 手戻り | 原因 | 最初にこうすべきだった |
|--------|------|----------------------|
| DELETEキー不具合 | 複数選択の削除ロジック漏れ | 「削除は単体・複数両対応で」と明記 |
| フォント二回り大きく | 初期の小さすぎるフォント | 初期指示で「業務用なので可読性重視」 |
| 3回の「初期セット」コミット | コミット戦略未定義 | 最初に「Conventional Commits形式で」 |
| GitHub Pages の手動設定 | 配布方法の後決め | 最初に「GitHub Pagesで公開前提」 |

## 5. プロジェクト開始テンプレート

```markdown

- 一言説明:
- 想定ユーザー:

- フレームワーク: (なし / React / Vue / etc.)
- サーバー: (不要 / Node.js / Python / etc.)

- 配布方法: (GitHub Pages / npm / Docker)

- エンティティ2: (フィールド一覧)

# 画面構成

# 優先機能（Phase 1で必須）
1.
2.
3.

# 将来機能（Phase 2以降）

# 非機能要件
- コミット規約: Conventional Commits
- ドキュメント: README.md + DEVELOPMENT.md
- テスト: (手動 / 自動)
```

## 6. AIとの協働で重要な原則

1. **小さく作って頻繁に確認する** — 100行ごとに動作確認
2. **ブラウザで見せる** — 動くものを見てからフィードバック
3. **「提案して」と聞く** — AIの知見を活用（例: ショートカット提案）
4. **enhance.txt → GitHub Issues** — 管理ツールは早めに決める
5. **引き継ぎを前提に書く** — `.agents/context.md` を最初から用意
6. **コミットは機能単位で** — 「初期セット」×3 ではなく機能名で

## 7. このプロジェクト固有のtips

- **Canvas アプリは状態管理が命** — `state` オブジェクトを最初に設計すべき
- **座標系は早めに確定** — world座標 ↔ screen座標の変換は全描画に影響
- **ビューモード（スクエア/クォーター）は後から追加が大変** — 初期設計に含める
- **Undo/Redo はスナップショット方式が最もシンプル** — 差分管理は複雑すぎる
- **localStorage は5MB制限** — 大規模データはJSON保存を併用する設計に
